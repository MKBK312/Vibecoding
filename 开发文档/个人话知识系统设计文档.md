# MindLink AI — 个人知识内化 RAG 系统设计文档

**日期：** 2026-05-07
**版本：** v1.0

---

## 1. 项目概述

**项目名称：** MindLink AI
**项目类型：** 本地个人知识内化 RAG 系统
**核心定位：** 帮助学生快速吸收 PDF 论文、技术文档、Obsidian/Markdown 笔记的本地 Web 应用

**核心亮点：**
- **精准引用溯源**：AI 回答时标注文件名 + 页码，可点击跳转原文
- **一键结构化摘要**：自动生成文档大纲，从"海量资料"快速转化为"个人知识大纲"
- **多源数据解析**：PDF 论文 + Obsidian/Markdown 笔记统一处理
- **流式对话体验**：参考来源卡片随问答折叠展开，边聊边查

**用户场景：**
- 纯本地运行（无网络依赖，隐私优先）
- 中小规模知识库（< 1000 篇文档）
- 以 PDF 论文 + Obsidian/Markdown 混合文档为主
- 本地 Web 服务形态（浏览器访问，可自定义扩展）

---

## 2. 技术选型

| 层级 | 技术选型 | 说明 |
|------|----------|------|
| 前端框架 | Next.js 14+ (App Router) | 全栈统一，API Routes 内置流式 SSE 支持 |
| 向量数据库 | ChromaDB | 本地零依赖，单文件运行，适合 <1000 篇文档 |
| Embedding 模型 | bge-small-zh (384维) | 中文效果好，本地运行，免费快速 |
| LLM | Anthropic Claude (云端) | 200K 上下文，结构化输出强，中文优秀 |
| PDF 解析 | pdf.js (浏览器端) / PyMuPDF (备选) | 网页端直接解析，避免大文件上传 |
| Markdown 解析 | 原生解析（marked / unified） | Obsidian 笔记直接读取 .md 文件 |

**架构模式：** Next.js 全栈（App Router + API Routes），单一代码库，部署简单。

## 4. 数据模型（ChromaDB Collections）

### 4.1 `documents` collection — 文档元数据

| 字段 | 类型 | 说明 |
|------|------|------|
| id | string (UUID) | 主键 |
| title | string | 文档标题 |
| source_type | string | pdf / markdown |
| file_path | string | 原始文件路径 |
| file_hash | string | 文件 MD5，用于去重检测 |
| metadata | dict | 作者、页数、创建时间等 |
| summary | string | AI 生成的结构化摘要 |
| chunk_count | int | 分割后的块数量 |
| created_at | datetime | 创建时间 |
| updated_at | datetime | 更新时间 |

### 4.2 `chunks` collection — 文本块

| 字段 | 类型 | 说明 |
|------|------|------|
| id | string (UUID) | 主键 |
| document_id | string | 所属文档 ID（用于 JOIN） |
| content | string | 原始文本内容（500-800字/块） |
| chunk_index | int | 块在文档中的顺序 |
| page_number | int | PDF 页码（非 PDF 文档为 null） |
| metadata | dict | { source_file, page_range } |

**检索时通过 document_id 关联**，还原完整文档上下文用于引用展示。

## 5. 核心流程

### 5.1 文档入库流程

```
文件拖拽上传
    │
    ▼
文件类型检测（PDF / Markdown）
    │
    ▼
文本提取
  · PDF → pdf.js 逐页提取文字 + 元数据
  · Markdown → 读取 .md 文件，解析 front-matter
    │
    ▼
智能分块
  · PDF：按页/段落分割，保留 500-800 字/块
  · Markdown：按 ## 标题或自然段落分割
    │
    ▼
bge-small-zh 生成向量（本地调用）
    │
    ▼
存入 ChromaDB（documents + chunks collections）
    │
    ▼
更新知识库状态（文档数、总页数、状态）
```

### 5.2 问答流程

```
用户输入问题
    │
    ▼
bge-small-zh 问题向量化（本地调用）
    │
    ▼
ChromaDB Top-K 语义检索（默认 K=5，可调节）
    │
    ▼
获取相关块（content + document_id + page_number）
    │
    ▼
构建 Prompt
  System: 你是学术研究助手，回答时必须标注来源 [文件名:页码]
  User: [问题] + [上下文片段]
    │
    ▼
Claude API 流式生成（SSE）
    │
    ▼
返回：AI 回答（带引用标注）+ 参考来源列表
```

### 5.3 结构化摘要流程

```
用户请求摘要（点击文档卡片或按钮）
    │
    ▼
获取该文档全部 chunks
    │
    ▼
构建摘要 Prompt
  System: 你是学术助手，为文档生成结构化摘要
  User: 请根据以下内容生成摘要：...
    │
    ▼
Claude API 生成（结构化格式：背景/方法/结论/要点）
    │
    ▼
更新 document.summary 字段
    │
    ▼
前端展示摘要卡片
```

## 6. API 设计（Next.js API Routes）

### 文档管理

| 方法 | 路径 | 说明 |
|------|------|------|
| POST | /api/documents/upload | 拖拽上传文件，返回 document_id |
| GET | /api/documents | 列出所有文档（含摘要、分块数） |
| GET | /api/documents/[id] | 获取单个文档详情 |
| DELETE | /api/documents/[id] | 删除文档（同时删除关联 chunks） |
| POST | /api/documents/[id]/reindex | 重新解析并索引文档 |

### 检索与问答

| 方法 | 路径 | 说明 |
|------|------|------|
| POST | /api/search | 向量语义检索，返回 Top-K 相关块 |
| POST | /api/chat/stream | 流式对话（SSE），返回回答 + 来源 |
| POST | /api/summary/[id] | 为指定文档生成结构化摘要 |

### 配置管理

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | /api/config | 获取当前配置（Top-K、模型状态等） |
| PUT | /api/config | 更新 Top-K 等运行时参数 |

## 8. 关键设计决策

### Q1: 为何选 ChromaDB 而非 PostgreSQL + pgvector？

< 1000 篇文档规模下，ChromaDB 无需额外数据库服务进程，单文件运行，零运维负担。pgvector 的 IVFFlat 索引在如此规模下优势不明显，反而增加部署复杂度。

### Q2: 为何 Embedding 本地 + LLM 云端？

Embedding 需要频繁调用（每次检索都要 encoding），本地 bge-small-zh 速度快且免费；LLM 效果差异明显，Claude 的 200K 上下文和结构化输出能力远超同等规模本地模型，且文档隐私由用户本地文件保障。

### Q3: 为何 Next.js 全栈而非 Next + FastAPI 分层？

1000 篇文档规模无需独立后端服务，Next.js API Routes 足够支撑，且单一代码库部署简单，内置流式 SSE 支持。FastAPI 分层更适合需要长时间异步任务（视频处理、大批量转录）的场景。

### Q4: PDF 解析用 pdf.js 而非服务端 PyMuPDF？

用户直接在前端上传 PDF 文件（通常 < 50MB），pdf.js 在浏览器端直接解析，无需上传大文件到服务端，减少网络开销和隐私风险。
